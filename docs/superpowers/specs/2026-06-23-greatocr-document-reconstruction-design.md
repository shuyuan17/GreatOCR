# GreatOCR 文档重建产品规格说明书

**英文名称：** GreatOCR Document Reconstruction  
**版本：** V0.3  
**日期：** 2026-06-30  
**状态：** 设计已确认，待实施计划  
**当前基线：** V1 可回退版本位于 `releases/v1`

---

## 1. 文档目的与修订说明

本规格定义 GreatOCR V2 的产品形态、用户流程、技术边界、安全规则和验收标准。V2 不再以 Codex Skill 作为主要产品入口，而是提供可在 Windows 10/11 便携运行的本机图形化应用。

V0.3 相对 V0.2 的主要变化：

- 产品入口改为“本机服务 + 浏览器图形界面”。
- 采用路线 2：先重构文档重建核心，再建设图形化页面。
- 支持选择 PDF 的部分页面处理，只上传选中页面。
- 将文档解析拆分为文字识别、版面分析、表格识别和可选智能校正等能力。
- MinerU 作为首个正式解析方案，并增加通用多模态实验适配器。
- 增加模型能力检测、敏感文件模型选择、失败后备选服务和页面级返工。
- 增加任务历史、串行队列、结果版本和 Windows 便携版交付。
- 翻译、总结和 Excel 首期仅保留扩展入口，不实现实际处理逻辑。

## 2. 产品目标

GreatOCR 面向需要把扫描或混合 PDF 转换为可编辑 Word 的业务用户，重点解决合同、董事会决议、财务资料和其他正式文档的文字、段落、图片、表格及页面方向还原问题。

V2 的核心目标：

1. 将单个 PDF 的全部页面或指定页面转换为可编辑 DOCX。
2. 明显改善 V1 的英文粘连、断行、页面方向、图片缺失和版面失真问题。
3. 允许用户选择解析服务，并安全配置自己的 API Key。
4. 对敏感文件明确展示数据流向，并在使用公网服务前逐任务二次确认。
5. 支持失败页面单独重试，并重新生成完整文档。
6. 为翻译、总结和 Excel 提取保留稳定扩展接口。
7. 以无需管理员权限的 Windows 便携版交付给普通用户。

成功标准不是与 PDF 像素级一致，而是：典型业务文档只需快速校对和少量微调，无需重新整体排版；任何未可靠还原的内容必须明确报告，不能静默遗漏。

## 3. 首期范围与边界

### 3.1 首期包含

- Windows 10/11 单用户本机应用。
- 选择单个 PDF 文件或输入本地路径。
- 本地预检页数、尺寸、旋转、加密和基础页面类型。
- 缩略图勾选页面和直接输入页码范围，例如 `3, 8-12, 26`。
- 超大 PDF 可跳过全量缩略图，直接输入页码范围。
- 默认把选中页面合并为一份 Word；可按页码区间分别输出。
- MinerU 正式解析适配器。
- 通用多模态实验适配器。
- 可选智能校正，默认关闭。
- 可编辑 DOCX、质量报告和结构化中间模型。
- 任务队列、任务历史、取消、阶段边界暂停、检查点恢复。
- 页面级失败、重试、替换和整份文档重新生成。
- 失败后询问用户是否改用其他已批准服务。
- Windows 便携版和完整源码工程。
- 翻译、总结和 Excel 的不可执行扩展入口。

### 3.2 首期不包含

- 本地大模型下载、安装或自动部署。
- 安装版、系统服务、开机启动项或管理员权限要求。
- Electron 桌面壳。
- 用户账号、多人协作、云端任务中心和企业权限系统。
- 翻译、总结和 Excel 的实际处理逻辑。
- PDF 与 Word 左右对照、拖拽区块和所见即所得编辑器。
- Word 内部原位修改；返工后从统一模型重新生成整份 DOCX。
- 自动安装、启动或控制 Microsoft Word。
- 与原 PDF 像素级一致的承诺。

## 4. 产品形态与技术选择

### 4.1 运行形态

便携目录包含启动程序和运行文件。用户解压后双击 `GreatOCR.exe`：

1. 启动仅监听 `127.0.0.1` 的本机后台。
2. 随机选择本机端口并生成临时访问令牌。
3. 自动打开默认浏览器显示 GreatOCR 页面。
4. 关闭 GreatOCR 后后台服务和临时令牌同步失效。

浏览器只是本机控制面板，不代表文件默认上传互联网。只有用户启动在线解析任务后，选中的页面才会发送给已确认的服务。

### 4.2 推荐技术栈

- 后端：Python + FastAPI。
- 前端：React + TypeScript；发布时编译为静态资源，用户无需安装 Node.js。
- 本地任务数据：SQLite。
- 凭据：Windows Credential Manager 或等价的当前用户安全凭据机制。
- PDF 与 DOCX：沿用 Python 文档处理生态，重构 V1 的模型映射和 Word 重建层。
- 打包：Windows 便携目录，不捆绑本地 AI 模型和完整浏览器内核。

### 4.3 发布体积原则

便携版不捆绑本地模型、Electron 或 LibreOffice。最终体积由 Python 运行时、PDF 渲染库和 DOCX 处理库决定；目标是保持在普通企业内部分发可接受的范围。实际体积在依赖锁定和首次构建后记录，不在规格阶段承诺固定数值。

未经数字签名的新程序仍可能触发 Windows SmartScreen 或企业安全软件提示。便携版可以避免管理员安装，但不能保证完全没有安全告警。

## 5. 页面结构与用户流程

### 5.1 任务中心

任务中心是默认首页，包含：

- 新建任务。
- 正在处理、等待、已完成、部分成功、失败和已取消任务。
- 文件显示名、选定页码、解析方案、进度、质量评级和输出位置。
- 打开任务详情、结果文件和输出目录。
- 重新处理失败页面或重新生成 Word。

首期同一时间只执行一个任务，其余任务排队，避免大型 PDF 同时处理造成资源竞争和供应商限流。

### 5.2 新建任务向导

#### 第一步：选择文件与页面

- 选择 PDF 或输入本地路径。
- 本地显示页数、文件大小、加密状态、页面方向和基础页面类型。
- 支持缩略图勾选与页码范围输入，两者保持同步。
- 页码以原 PDF 的一基页码显示。
- 解析前校验页码范围、重复页和空选择。

#### 第二步：安全与操作

- 回答文件是否敏感。
- 选择“转为可编辑 Word”。
- 翻译、总结和 Excel 显示为“即将支持”，不能启动。

#### 第三步：解析方案

- 默认选择 MinerU。
- 显示方案是否支持文字、坐标、版面、表格、图片和公式。
- 智能校正默认关闭，由用户主动启用。
- 能力不足的服务明确标记为“文字优先 Word”。
- 敏感文件选择公网或未验证服务时显示风险标记。

#### 第四步：输出与确认

- 选择输出目录。
- 默认合并全部选中页面；可按用户输入的页码区间分别输出。
- 显示将上传的具体页码、服务、接口地址、可能的备选服务和留存策略。
- 敏感文件必须完成二次确认才能开始。

### 5.3 任务详情

任务详情显示预检、拆页、上传、解析、标准化、可选校正、Word 生成、质量检查等阶段，以及已用时间、预计剩余时间和可读错误信息。

“暂停”只在阶段边界生效。外部 API 已提交的任务未必能远程暂停或撤销，界面必须明确说明。任务失败后可以从最近成功检查点继续。

### 5.4 设置中心

- 管理正式解析服务、实验性多模态模型和文本校正模型。
- 配置接口地址、模型名称和 API Key。
- 显示连接测试和能力测试结果。
- 显示“已配置”和密钥末尾少量字符，不显示完整密钥。
- 删除服务配置时同时删除对应本地凭据。
- 高级设置包含语言、页眉页脚、分页、数字日期格式和中间文件留存。

## 6. 文档处理数据流

标准流程：

1. 本地预检 PDF。
2. 解析并确认页面选择。
3. 为选中页面建立原始页码映射。
4. 本地生成仅包含选中页面的临时子 PDF；需要时可按页生成单页任务。
5. 调用用户确认的解析方案。
6. 将供应商结果标准化为统一文档模型。
7. 执行规则校正和可选智能校正。
8. 根据结构、坐标、方向和资源生成 DOCX。
9. 执行完整性、关键字段、图片和 DOCX 结构检查。
10. 生成质量报告和任务版本。
11. 按安全与留存策略清理临时文件。

未选页面不得发送给解析服务。质量报告中的页码使用原 PDF 页码，而不是临时子 PDF 页码。

## 7. 统一文档模型 V2

统一文档模型是 Word 生成、质量检查、页面返工和未来翻译的权威来源。Markdown 只用于阅读和排错，不能作为 Word 重建的唯一来源。

### 7.1 核心实体

- `Document`：文档指纹、来源显示名、选页规则、供应商记录、输出版本。
- `Page`：稳定 ID、原始页码、任务内页码、原始宽高、有效宽高、旋转、页面类型和处理状态。
- `Block`：标题、段落、列表、表格、图片、页眉、页脚、页码、公式等结构。
- `TextSpan`：原值、当前值、语言、置信度、字体特征、坐标和修改记录。
- `Table` / `TableCell`：行列、跨行跨列、边框、对齐、坐标、置信度和降级状态。
- `Asset`：内容指纹、相对路径、类型、来源页码、坐标、尺寸和裁剪信息。
- `Issue`：页码、关联结构、严重程度、原因、片段和建议。
- `ProviderTrace`：每页使用的服务、模型、处理时间和重试记录，不包含密钥。

### 7.2 坐标与方向

- 坐标在进入统一模型时归一化为相对于页面宽高的比例坐标。
- 同时保留供应商原始坐标和坐标单位，便于排错。
- 页面旋转在生成内容前标准化，计算视觉上的有效宽高。
- Word 页面方向依据视觉有效方向设置，不能直接使用未处理的 PDF MediaBox 宽高。

### 7.3 稳定标识与页面替换

页、区块、表格和文本片段使用稳定 ID。页面重新解析时只替换目标页的数据，未受影响页面尽量保持 ID 不变。这样可以支持失败页面重试、质量问题追踪和未来翻译映射。

## 8. 解析能力与供应商适配

### 8.1 能力角色

- `TextRecognizer`：文字、行、单词、坐标和置信度。
- `LayoutAnalyzer`：标题、正文、分栏、页眉页脚、图片区域和阅读顺序。
- `TableRecognizer`：表格行列、单元格、合并关系和表头。
- `DocumentParser`：可同时实现上述多项能力的一体化服务。
- `TextReasoner`：断行、空格、结构复核和风险检测。
- `Translator`：未来翻译能力。

用户界面可把这些能力组合显示为一个“解析方案”，但内部不能假设每个 OCR 服务都具备版面和表格能力。

### 8.2 正式适配器

MinerU 是首个正式解析适配器。特殊文档服务如果具有自定义上传、异步轮询、ZIP 下载或专有 JSON，必须实现专用适配器后才能标记为正式支持。

供应商适配器按 API 协议和结果结构编写，不必为同一协议下的每个具体模型重复编写。

### 8.3 通用多模态实验适配器

用户可配置兼容的多模态模型接口。GreatOCR 将页面图片和结构化指令发送给模型，要求其按 GreatOCR JSON Schema 返回文字、区块类型、坐标和置信度。

启用前必须执行样例页能力测试：

- 能否接收图片。
- 能否稳定返回合法 JSON。
- 是否提供文字、坐标、版面、表格和图片区域。
- 是否通过字段完整性和页码映射检查。

测试结果分为：完整解析、文字优先、仅可校正和不兼容。实验性适配器必须明显标记，不能冒充正式稳定支持。

### 8.4 完全未知的新服务

用户可以填写通用协议兼容模型，但完全不同的专有 OCR API 仍需要开发新适配器。首期不允许普通用户安装未知 Python 插件，避免任意代码执行风险。

后续可评估千问、Kimi、豆包、DeepSeek 等服务；是否作为完整解析器、校正器或翻译器，以当时实际 API 和能力测试为准，不按品牌预先假定。

## 9. 智能校正

智能校正默认关闭。未配置或校正失败时，系统仍使用解析结果生成 Word。

校正范围包括：

- 明显的英文单词粘连和缺失空格。
- OCR 造成的错误断行、段落切分和跨行连接。
- 标题层级、列表和阅读顺序的低置信复核。
- 对低置信页面或局部区域的二次判断。

校正必须遵守：

- 不为提高语言流畅度而改写原文。
- 金额、日期、账号、税号、编号、人名和公司名等关键字段无可靠依据不得修改。
- 每次自动修改保存原值、当前值、依据、来源和置信度。
- 低置信建议进入质量报告，不自动采纳。
- 对外发送的页面或区域必须受当前任务的数据流确认约束。

## 10. Word 重建规则

### 10.1 页面方向与分节

- 根据标准化后的视觉方向设置 Word 页面宽高和方向。
- 混合横竖页面使用 Word 分节处理。
- 新分节必须正确处理页眉页脚链接关系。
- 默认尽可能保持原分页；偏差写入质量报告。

### 10.2 版面与阅读顺序

- 利用相对坐标识别段落邻接、双栏、缩进和区域层级。
- 不能只按供应商数组顺序追加段落。
- 可编辑性优先时使用流式 Word 元素；无法可靠流式还原时允许局部降级并报告。
- V2 不承诺复杂自由定位页面的像素级还原。

### 10.3 文字与段落

- 根据坐标间距、语言和标点规则合并行与段落。
- 英文合并时处理单词间空格和行末断词。
- 中英文混排采用可配置字体回退，不自动下载字体。
- 页眉页脚进入真正的 Word 页眉页脚区域，不作为普通正文段落。

### 10.4 图片、签名和印章

- 资源使用内容指纹和任务内相对路径，禁止依赖旧电脑绝对路径。
- 图片尺寸依据原页面占比和可用页面宽度计算，不使用统一固定尺寸。
- 优先使用内联图片保证跨 Word/LibreOffice 稳定性。
- 签名和印章作为图片保留。
- 供应商未导出资源时，尝试从原 PDF 对应坐标裁剪；仍失败则生成明确 issue。

### 10.5 表格

- 高置信表格生成真实 Word 表格。
- 保留行列、合并单元格、边框、对齐、底色和合理列宽。
- 低置信或结构冲突时降级为原图，不生成看似可编辑但结构错误的表格。
- 表格降级必须记录页码和原因。

## 11. 失败处理、备选服务与页面返工

### 11.1 服务失败策略

每个任务提供：

1. 不切换：当前服务失败后停止。
2. 失败时询问：默认策略；列出能力满足要求的已配置服务，由用户选择。
3. 自动选择：只从用户事先批准的备选服务池中选择。

GreatOCR 选择备选服务时必须：

- 只使用能力满足当前任务的服务。
- 只发送失败页面，不重复发送成功页面。
- 不使用未配置、未批准或未展示的数据目的地。
- 在质量报告中记录每页实际使用的服务。
- 敏感任务开始前明确展示并批准所有可能使用的备选服务。

首期只有一个正式解析适配器时，界面和策略可存在，但在没有可用备选服务时明确提示不能切换。

### 11.2 单页失败与重新拼合

- 单页失败不阻止其他页面继续处理。
- 初次输出可使用原页面图片作为失败页占位，并标记“部分成功”。
- 用户可只重新解析失败页或指定页。
- 成功后替换统一文档模型中对应页面的数据。
- 其他成功页面不重新调用解析 API。
- 从更新后的统一模型重新生成整份 Word 和质量报告。
- 原始页码、用户分区和页面顺序保持不变。

### 11.3 结果版本

每次重建生成不可混淆的版本，例如：

- `result-v1.docx`：首次结果。
- `result-v2.docx`：返工后的完整结果。
- `result.docx`：当前最新结果副本或稳定入口。

质量报告和任务清单记录版本来源、替换页面和生成时间。保留数量由设置控制，默认保留最近几个版本，不静默覆盖唯一可回退结果。

## 12. 安全与隐私

### 12.1 API Key

- 保存在 Windows 当前操作系统用户的安全凭据区。
- 不写入便携目录、项目配置、SQLite、任务目录、日志、异常消息或报告。
- 前端提交后不长期保存完整密钥。
- 页面只显示“已配置”和末尾少量字符。
- 删除服务配置时同时删除本地凭据。
- 便携目录复制给其他用户时不携带原用户密钥。

### 12.2 普通文件

- 可以使用已配置和已确认的在线服务。
- 开始前显示供应商、上传页码和留存策略。
- 任务历史可保存文件显示名、选页、模型、状态和输出位置。
- 可保留必要中间数据用于恢复和返工。

### 12.3 敏感文件

敏感文件由用户确认分类。系统可以提示风险，但不能替代企业合规判断。

- 进入明显的敏感模式。
- 公网调用默认不能直接启动。
- 显示供应商、接口地址、具体上传页码和发送内容。
- 用户必须逐任务二次确认。
- 确认不跨任务复用。
- 只上传选定页面。
- 上传成功后删除临时拆分页；需要重试时从本地原 PDF 重新生成。
- 默认只保留最终 Word 和质量报告，清理解析缓存和页面图片。
- 敏感任务历史默认隐藏原文件名，显示匿名任务名称；用户可主动保留文件名。
- 不允许静默切换供应商。

### 12.4 本机网页安全

- 只监听回环地址，不开放局域网访问。
- 每次启动使用随机端口和临时令牌。
- 校验请求来源和令牌，防止其他网页调用本机服务。
- 关闭程序后令牌失效。
- 文件访问限制在用户显式选择的输入、输出和任务目录内。

## 13. 任务、存储与留存

### 13.1 任务队列

- 同时只运行一个主任务。
- 等待任务按创建顺序排队，用户可以取消等待任务。
- 任务阶段状态为 pending、running、paused、succeeded、partial、failed、cancelled 或 skipped。
- 每个主要阶段写入检查点。

### 13.2 SQLite 记录

SQLite 只保存任务元数据和配置索引，不保存 API Key。正文内容和完整供应商响应保存在任务目录，并受留存策略控制。

### 13.3 建议目录

普通任务：

```text
task-dir/
├─ result.docx
├─ result-v1.docx
├─ quality-report.docx
└─ intermediates/
   ├─ document.json
   ├─ task-manifest.json
   ├─ quality-report.json
   ├─ provider-raw/
   └─ assets/
```

敏感任务默认完成后只保留：

```text
task-dir/
├─ result.docx
├─ result-v1.docx
└─ quality-report.docx
```

## 14. 异常与降级处理

- PDF 不存在、损坏、加密或页码无效：上传前停止并说明原因。
- 网络中断、限流或临时错误：有限次数重试，仍失败则保存检查点。
- 服务永久失败：按任务的失败策略停止、询问或选择已批准备选服务。
- 智能校正失败：跳过校正并继续，报告中说明。
- 单页失败：其他页面继续；失败页原图占位并允许后续返工。
- Word 生成失败：保留统一模型；再次生成不重复调用解析 API。
- 图片缺失：内容指纹重定位、原 PDF 裁剪、明确 issue 依次降级。
- 用户取消：说明外部服务已提交任务可能无法远程撤销。
- 任何部分成功都不得静默省略页面或内容。

## 15. 质量报告与自动检查

每个任务至少检查：

- 所选页面是否全部对应输出或明确失败占位。
- 原始页码映射是否正确。
- 页面横竖方向是否正确。
- 图片、签名、印章、表格是否输出或有明确降级记录。
- 是否存在明显英文单词粘连和异常断行。
- 关键字段是否被无依据修改。
- DOCX 包结构是否完整并可由标准库重新读取。
- 页眉页脚是否进入正确区域。
- 输出和日志中是否泄露 API Key。
- 每页实际使用的供应商和重试记录是否可追溯。

面向用户的正式报告为 `quality-report.docx`，结构化数据可保存在 `quality-report.json`。敏感模式按留存策略清理结构化中间报告。

## 18. 最终验收标准

- 竖版 PDF 不再因忽略旋转而生成横版 Word。
- 选定页面全部进入输出，未选页面不上传。
- 图片和印章不因旧电脑绝对路径而丢失。
- 英文粘连、断行和段落切分相对 V1 明显改善。
- 坐标和版面信息实际参与 Word 重建。
- 失败页面可以单独重试并重新生成完整文档。
- 供应商切换受用户批准和敏感文件规则约束。
- API Key 不出现在任务数据库、目录、日志或报告中。
- 便携版可以在无 Python、无 Node.js、无管理员权限的 Windows 10/11 电脑运行。
- 翻译、总结和 Excel 可以复用相同的任务、页面选择、供应商和安全架构。

## 19. 主要风险与设计取舍

- PDF 是固定页面模型，DOCX 是流式排版模型，完全可编辑与像素级一致不能同时保证。
- 通用视觉模型的结构化输出可能不稳定，因此只能作为实验方案并经过能力测试。
- 特殊 OCR 服务必须维护适配器，不能承诺任意 API 即插即用。
- 敏感文件允许用户确认后调用公网 API，但最终合规责任仍由用户或其组织判断。
- 不捆绑 LibreOffice 和本地模型可以控制体积，但运行时无法完成与开发环境相同的逐页视觉渲染验证。
- 便携版减少安装摩擦，但未签名程序仍可能被企业安全软件拦截。
- V2 首期优先完成可靠的任务式处理，不扩展为完整文档编辑器。
