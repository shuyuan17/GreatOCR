## 2. 用户会困惑的地方

### 2.1 "fake-default" 含义模糊

用户看到下拉列表中 `fake-default (已配置)`，会疑惑 "fake 是什么？它是真的能用还是假的？"

**建议**：在 UI 上标注 `🛠️ 模拟模式（离线测试，无需 API Key）`，或使用 `local-test`、`testing-mode` 等更直观的名称。

### 2.2 前后端需要两个终端

虽然 `serve.py` 一键启动了后端，但前端仍需在另一个终端手动启动。一个完整的"开始使用"流程需要：
1. 开一个终端运行后端
2. 开另一个终端运行前端
3. 浏览器访问 `localhost:5173`

对非技术用户来说，这是两个额外的概念（终端、端口号）。

### 2.3 下载链接难以发现

结果文件和报告文件的下载链接字号为 `0.75rem`（约 12px），颜色为 `#999`（浅灰）。用户可能根本注意不到这些链接，从而认为"结果无法下载"。

### 2.4 "打开输出文件夹"无反馈

点击 📁 按钮后，如果后端不支持或操作失败，catch 块为空，用户得不到任何反馈。点击后可能认为"按钮坏了"。

### 2.5 输出目录需手动输入

无原生文件夹选择对话框，用户需要手动输入完整路径（如 `D:\my-ocr-results`），容易输错。

### 2.6 导航无激活态指示

当前页面在导航栏没有高亮显示，用户可能不清楚自己当前在哪个页面。

### 2.7 后端错误码无中文解释

当 OCR 失败时，用户看到的是 `{"code": "CREDENTIAL_NOT_CONFIGURED"}` 之类的错误码，而非"请先配置 API Key"这样的中文提示。

## 5. 是否建议进入"一键启动/打包"阶段

### 结论：**不建议立即进入打包阶段**

### 理由

**必须修复的 3 个 P0 问题**（编码乱码、静默错误、download_path 回退 #）可能导致用户在核心流程中遇到不可理解的错误或功能不可用。
