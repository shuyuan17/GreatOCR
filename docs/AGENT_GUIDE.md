# AGENT GUIDE

## 文档目的

这份文档用于统一 GreatOCR 项目中的 AI 协作方式，帮助产品、技术负责人和 Coding Agent 之间形成稳定、低风险、可追踪的工作流程。

## 角色分工

### 产品/业务

- 提出需求
- 决定产品方向
- 测试功能
- 管理 GitHub 与任务优先级

默认前提：

- 不假设产品同学能直接阅读复杂代码或架构
- 输出应尽量使用产品视角和可理解的语言

### 技术负责人

- 拆解需求
- 制定任务计划
- 审查 Agent 产出
- 判断哪些文件应该提交
- 解释 Git、架构和实现方案
- 控制开发节奏

边界：

- 不要求一次性写完全部代码
- 重点是把任务拆小、把风险讲清楚、把质量守住

### Coding Agent

- 只完成明确任务
- 不擅自扩大范围
- 不重构整个项目
- 不修改无关文件

完成任务后必须说明：

- 修改了哪些文件
- 为什么修改
- 如何启动
- 如何测试
- 是否还有未完成事项或风险

## 工作原则

### 一次只做一个任务

- 每次任务控制在 30 分钟到 2 小时为宜
- 一个任务只解决一个明确问题
- 完成后及时整理结果，不把多个不相关需求混在一起

### 小步快跑

优先级顺序：

1. 可以运行
2. 可以测试
3. 可以提交 Git
4. 再考虑 UI 美化、重构、性能优化

### 不扩大影响面

禁止行为：

- 一次完成整个项目
- 重构整个仓库
- 修改无关模块
- 删除已有功能
- 大规模替换技术栈

## Git 工作流

### 分支规范

- `main` 永远保持稳定可运行
- 新功能必须在独立分支开发

推荐命名示例：

- `feature/frontend-shell`
- `feature/task-center`
- `feature/settings`
- `feature/pdf-preview`

### 每完成一个任务后

先查看改动：

```powershell
git status
```

提交原则：

- 优先只提交本次任务相关文件
- 只有当确认本次所有修改都应该提交时，才使用 `git add .`

常见流程：

```powershell
git add <文件名>
git commit -m "描述本次任务"
git push
```

第一次推送新分支：

```powershell
git push -u origin <分支名>
```

## Agent 开始工作前必须阅读

1. `README.md`
2. `PROJECT_STATUS.md`
3. `docs/reports/RUN_REPORT.md`（如果存在）
4. `docs/reports/FRONTEND_REPORT.md`（如果存在）
5. 当前任务说明

要求：

- 不要直接开始写代码
- 先理解当前阶段、边界和已有结论

## Agent 完成工作后必须输出

- 修改了哪些文件
- 为什么这么改
- 如何启动
- 如何测试
- 是否存在风险
- 建议下一步

必要时更新以下文档：

- `PROJECT_STATUS.md`
- `docs/reports/RUN_REPORT.md`
- `docs/reports/FRONTEND_REPORT.md`
- `docs/reports/VALIDATION_REPORT.md`

## 沟通风格

- 尽量使用产品经理能理解的语言
- 少用不必要的术语
- 每次说明“为什么”
- 如果涉及 Git、流程或架构，不只给命令，还要解释原理

## GreatOCR 项目的推荐协作方式

当收到一个新需求时，建议按下面流程推进：

1. 先分析需求
2. 把需求拆成多个小任务
3. 为 Agent 生成明确任务说明
4. Agent 执行单个任务
5. 技术负责人 Review
6. 整理提交内容并提交 Git
7. 再进入下一任务

## 文档维护建议

- `README.md`：面向任何第一次接手项目的人，解释项目是什么、现在到哪一步了
- `PROJECT_STATUS.md`：记录阶段进度、能力边界、已知问题和下一步
- `docs/TASK_BOARD.md`：记录任务推进顺序和状态
- `docs/reports/`：存放运行、前端、验证等过程报告

## 最终目标

GreatOCR 不只是要把功能做出来，还要形成一套可持续的 AI 协作开发方式，让产品、技术负责人和 Agent 都能在同一套规则下稳定推进。
